Q1: What are the key architectural components to create a chatbot based on LLM? Please explain the approach on a high-level.
Answer: "In building my chatbot, I structured the system into three distinct functional layers to ensure the responses were both accurate and scalable:
1. The Orchestration Layer (Django) I utilized Django as the primary backend to handle the flow of data. Instead of just serving web pages, this layer acts as the controller—it captures the user's input, manages the session history, and executes the logic to sanitize the query before it interacts with the AI models.
2. The Cognitive Layer (Gemini Integration) For the reasoning engine, I integrated the Gemini API. This layer is responsible for natural language understanding. A key part of my implementation here was designing a 'System Prompt' that instructs the model on how to behave, ensuring consistent and relevant answers based on the context provided by my backend.
3. The Retrieval Layer (RAG with FAISS) To prevent the model from hallucinating, I implemented a Retrieval-Augmented Generation (RAG) pipeline using FAISS. I converted my custom dataset into vector embeddings. Now, before the chatbot answers, it first searches this vector store for the most relevant facts and feeds them to Gemini, ensuring the final answer is grounded in my specific data."

Q2: Please explain vector databases. If you were to select a vector database for a hypothetical problem, which one will you choose, and why?
Answer: "To put it simply, while a standard SQL database queries for exact keyword matches, a Vector Database queries for conceptual similarity. It stores data as high-dimensional vectors (embeddings). This allows the system to find data that 'means' the same thing as the user's query, even if the phrasing is completely different.
Hypothetical Problem: Let's assume I am scaling up my Cervical Cancer Detection project to include a 'Research Assistant' for doctors. This system needs to index millions of oncology papers and allow doctors to ask complex treatment questions.
My Selection: Pinecone While I used FAISS in my personal project because it is efficient for local, static datasets, for this production-level medical problem, I would select Pinecone.
My Reasoning:
1. Scalability: FAISS runs in-memory, which becomes a bottleneck if the dataset exceeds the server's RAM. Pinecone is a managed, serverless solution that handles scaling automatically.
2. Real-Time Updates: In the medical field, new research is published constantly. Pinecone handles real-time data ingestion and re-indexing much better than rebuilding a static FAISS index every time a new paper is added.
3. Developer Experience: Since I am already familiar with Python and LangChain, Pinecone’s native integration would significantly speed up the development of the pipeline."
